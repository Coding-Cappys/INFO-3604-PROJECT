# Deploying To Render

This guide covers the recommended Render setup for this repository.

The simplest and most reliable deployment path is:
- deploy the Flask app as the main web service
- use a Render PostgreSQL database
- do not rely on SQLite in Render

## Recommended Architecture

Deploy:
- one Render PostgreSQL instance
- one Render Python web service for the Flask app

Skip initially:
- the TypeScript API server

The Flask app contains the main user-facing workflows and is the best first deployment target.

## Option 1: Blueprint Deploy With `render.yaml`

This is the easiest route.

### What `render.yaml` creates

- a free PostgreSQL database named `uwi-festival-db`
- a free Python web service named `uwi-festival`

### Steps

1. Push this repo to GitHub.
2. In Render, click `New` -> `Blueprint`.
3. Connect your repository.
4. Render will detect [render.yaml](/home/jgb/Documents/uwi/yr3sem2/info3604/test/Flask-Jinja-Sqlalchemy/render.yaml).
5. Review the generated services.
6. Click deploy.

### Current Flask service settings

Build command:

```bash
pip install -r requirements.txt
```

Start command:

```bash
cd artifacts/uwi-festival && gunicorn --bind 0.0.0.0:$PORT wsgi:app
```

Environment variables:
- `DATABASE_URL` from the Render Postgres instance
- `SESSION_SECRET` auto-generated by Render

## Option 2: Manual Render Setup

Use this if you do not want to use Blueprints.

### Step 1: Create Postgres

In Render:
1. Click `New` -> `PostgreSQL`
2. Name it `uwi-festival-db`
3. Choose the free plan
4. Wait for provisioning

### Step 2: Create Web Service

In Render:
1. Click `New` -> `Web Service`
2. Connect the repo
3. Set:

- Environment: `Python 3`
- Root Directory: leave blank
- Build Command:

```bash
pip install -r requirements.txt
```

- Start Command:

```bash
cd artifacts/uwi-festival && gunicorn --bind 0.0.0.0:$PORT wsgi:app
```

### Step 3: Add Environment Variables

Add:
- `DATABASE_URL` = internal connection string from your Render Postgres instance
- `SESSION_SECRET` = a long random secret

Do not set:
- `PORT`

Render provides `PORT` automatically.

## Why This App Works On Render

The Flask app:
- creates tables at startup with `db.create_all()`
- seeds demo data only if the users table is empty

That means:
- first deploy initializes the schema
- first deploy inserts demo records
- later cold starts reconnect to the same Postgres DB
- seed data should not duplicate on every restart

Relevant code:
- [__init__.py](/home/jgb/Documents/uwi/yr3sem2/info3604/test/Flask-Jinja-Sqlalchemy/artifacts/uwi-festival/app/__init__.py)

## Important Render Caveats

### 1. Do not use SQLite on Render

Your Flask app falls back to SQLite if `DATABASE_URL` is missing.

That is fine locally, but not for Render, because the filesystem for free web services is not persistent in the way you need for app data.

Always set `DATABASE_URL` on Render.

### 2. Free web services sleep

Render free web services spin down after inactivity and wake on the next request.

This is usually fine for a demo app, but:
- first request after idle can be slow
- users may see a startup delay

### 3. Free Postgres is not permanent

Render free Postgres is suitable for demo/testing, but you should treat it as temporary infrastructure.

Do not use it for important long-term production data without understanding Render’s current retention/expiration rules.

## Troubleshooting

### App boots but shows database errors

Check:
- `DATABASE_URL` is set
- Postgres service is healthy
- Flask logs in Render show successful startup

### App deploys but gives 500s

Check the web service logs in Render for:
- import errors
- database connection errors
- template rendering errors

### App works locally but not on Render

Most likely causes:
- missing `DATABASE_URL`
- missing `SESSION_SECRET`
- trying to use SQLite on Render
- wrong start command
